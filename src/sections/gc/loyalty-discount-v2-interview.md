Задача на **написание кода в IDE** кандидата. Расчёт: **~60 минут**.

**Базовая часть** + **усложнение** (асинхронный сервис скидок, ретраи).

- **15 грейд:** база.
- **16 грейд:** усложнение.

Документация: **Секция general coding Java (экс livecoding / экспериментальная)**; в начале — **#rekomendaciipoprovedeniju**.

## Задача

Написать код для **расчёта и применения скидок** на корзину (процентная или фиксированная скидка на сумму корзины, распределение по позициям **пропорционально цене**).

## FAQ (вопросы кандидата)

| Вопрос | Ответ |
|--------|--------|
| Как округлять? | **Математическое** к копейкам: строго **меньше** 0.5 копейки → вниз, **не меньше** 0.5 копейки → вверх. |
| Валюта? | Только **рубли**, **копейки**; валют нет. |
| Скидка настроена для всех? | **Нет** — у части пользователей скидки нет → корзина **без** изменений (или явный «0%» — по контракту). |
| Мутировать корзину или копию? | **Обсудить** плюсы/минусы; выбор за кандидатом. За содержательное обсуждение — **плюс в карму**, отметить в результатах секции. |
| Сразу писать комментарии и доки? | **Не обязательно** (мало времени); хорошо, если спросил. |

**Важно:** два вида скидки — по сути **один алгоритм распределения**; разница только в том, как получить **общую сумму скидки** на корзину.

## Как проводить и что фиксировать

### Важно

- **API классов** придумывает **кандидат**. При советах **не** давать готовых рекомендаций по выбору — спросить про **плюсы/минусы** и предложить **выбрать самому**.

### В процессе

- Следить за временем: не уходить в **излишнюю сложность**, мягко **упрощать**.
- Если взял **`double`** для денег — **спросить почему**; если **сам** перейдёт на подходящий тип — **баллы не снимать**.
- После **первого** теста на happy path — предложить **остальные сценарии оформить комментариями**, потом реализовать по возможности (важно увидеть **мышление по кейсам**).
- Делать **заметки** (см. ниже).
- Ошибки в коде **не комментировать вслух** — пусть ловят **тесты**.
- Если не покрыты **округление**, **некорректная скидка**, **ошибки** — **наводящие вопросы** → в крайнем случае **подсказать сценарий** (**−1** балл в соответствующем критерии). Ожидается **тест + правка кода**.

### Что записать в результат секции

- **Тайминг:** выдали задачу → начал код → модели данных → расчёт скидки → начал тесты → тесты готовы → тесты зелёные; время и суть **подсказок**; для усложнения — те же этапы.
- **Вопросы кандидата** и ответы.
- **Поиск** (что искал и зачем).
- **Вопросы интервьюера** и ответы кандидата.
- **Код и тесты** — в интервьюшницу.
- Любые **сигналы** (+/−).

После достаточного объёма кода — попросить **скопировать код + тесты** в интервьюшницу.

## Усложнение

```text
Настройки скидок лежат во внешнем сервисе (БД, HTTP и т.д.).

Доступ к сервису асинхронный.
Сервис новый, и может возвращать ошибки, нужно уметь их переживать.
В случае иных ошибок можно вызвать повторно (не более 1 ретрая)
Каждый вызов не дольше 1 секунды.

Если после 1 ретрая не удалось узнать настройки пользователя - скидку посчитать не можем. Считаем, что скидок нет.

Нужно адаптировать код к изменениям.
```

### Подсказки интервьюеру

- Публичный API расчёта — **синхронный или асинхронный** — решает кандидат.
- Моки: **Mockito** / **Moq** или **ручная** реализация интерфейса — ок.
- В тестах — **дождаться** завершения async перед assert; ошибки async — обработать в коде и проверить в тестах.

**Когда выдавать:** сразу с базой, если тянет; иначе отдельно (**3–5 мин** не думает / код раздувается).

Результат усложнения — **отдельная** задача в интервьюшнице.

## Минимальное решение (acceptance)

1. **Корректное распределение** скидки по алгоритму (пропорционально цене линии).
2. **Валидация** процента скидки в **[0, 100]**.
3. **Округление** до копеек по оговорённым правилам.
4. Погрешность **суммы скидок по строкам** на **копейки** из‑за округлений — **не критична** (типично добить последней позицией / остатком).

### Требование на 15 грейд

Отдельных порогов нет — смотреть **баллы качества** в tear-off.

### Усложнение (acceptance)

- Ошибки **вызова API** обработаны по цепочке до внешнего контракта.
- **Ретрай** (не более **одного**) реализован корректно.

## Деньги и округление (рубрика)

### Хранение и расчёт

| Подход | Оценка |
|--------|--------|
| **BigDecimal** (Java), scale 2 | Хорошо |
| **decimal** (C#) + округление до копеек | Хорошо |
| **long** копейки + **double** в промежуточных | Нормально |
| **long** копейки + **целочисленная** схема (например `(price * (100 - p) + 50) / 100`) | Хорошо |
| **int** копейки | Как long, но **−0.5** за проектирование без внятного обоснования запаса по сумме |
| **double** | Только с **явным** округлением до копеек; **0** за проектирование по умолчанию |

### Округление

- Обязательно до **копеек** — иначе решение **не засчитывать**.
- Желательно **математическое** (0.5 вверх). Другое — возможно со **штрафами** по договорённости.
- Если кандидат **не спрашивает** и округляет «как получилось» — сказать правило, **−0.5** за проектирование; **добиться** корректности; если не вышло — решение можно **принять** с пометкой.

## Примеры тестов: основная часть

**Ошибки / валидация:**

- процент **вне** [0, 100];
- **предельные** суммы / границы.

**Happy path:**

- у пользователя есть скидка — **корректно** отражена в позициях корзины.

**Дополнительно:**

- **нет** скидки — корзина без изменений (или эквивалент);
- **округление**: кейс с **> 2** знаков после запятой в промежуточном результате — минимум **2** теста на математическое округление к копейке;
- неверный % → **ошибка**;
- при проверках на **null** — хотя бы **один** тест, остальное можно **комментариями**.

---

## Примеры тестов: усложнение

- **Транспортные** ошибки сервиса скидок — цепочка до публичного контракта.
- **Логические** ошибки сервиса, если валидация **внутри** сервиса.
- Happy path из базы — **адаптировать** под async.
- Ошибки сервиса + **ретрай** / итог «как без скидки» после неудачи.
