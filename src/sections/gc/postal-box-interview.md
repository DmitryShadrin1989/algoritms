# Разработка постамата

Задача на **код в IDE** по правилам **general coding** (внутренняя методичка секции).

**Можно использовать всё, кроме AI-ассистента.**

**Время:** ориентир **~1 час**.

---

## Постановка (текст для кандидата)

```text
Постамат - автоматическая станция приёма/выдачи посылок.
В маркете формируются заказы, и хочется добавить возможность получения через постамат.
Запускаем MVP: небольшая аудитория пользователей, несколько постаматов в Москве.
При заказе пользователь сможет выбрать, что хочет получить заказ в постамате.

В рамках задачи нужно реализовать код для MVP решения:
- курьер привозит заказ и пробует положить его в ячейку, указывая номер заказа.
  Постамат сам выбирает ячейку и возвращает в ответ. Она откроется вызывающим этот метод кодом.
- после того, как заказ положили в ячейку, пользователю отправляется СМС c кодом получения.
  Заказ будет ждать вечно
- в случае любых ошибок - курьер забирает заказ назад и попробует положить заказ в ячейку на следующий день (для MVP это ок)
- пользователь может получить заказ по коду выдачи из СМС.
  При вводе кода выдачи постамат должен вывести на экран текст "ваш заказ ХХХ в ячейке YYY", ячейка откроется сама.

Ограничения:
- все ячейки одного размера, но их может быть разное количество, зависит от конкретного постамата
- один заказ - одна коробка, она влезает в ячейку
- ячейки каждого постамата пронумерованы
- каждый постамат сам хранит своё состояние

Для отправки сообщения пользователю надо использовать клиент UserNotificationApi / IUserNotificationApi.
```

**Алгоритм** простой; акцент на **организации кода**, **сигналах качества**, **косяках**.

- Предложение **БД** → для MVP **только память** постамата; персистентность **не обязательна**.
- **API** уведомлений можно мыслить как **очередь**; ошибка возможна при **отправке** — её **нужно** обработать (если кандидат не заложил — можно **подсказать**, за подсказку по смыслу базовой части **не штрафуют**, см. Hire III).

## Усложнение: генерация кода выдачи

По практике — **сильное** усложнение.

- **Шестизначный** код (допустимы **ведущие нули**, т.е. по сути 000000–999999 или согласованный диапазон).
- **Уникальность** среди уже выданных кодов.
- **Достаточно случайный** источник: **`SecureRandom` / `RandomNumberGenerator`** и т.п.; **счётчик** или **хеш от orderId** как единственный источник — **не** принимать как решение усложнения.
- **Не подсказывать** конкретный класс — кандидат выбирает сам.

Выдавать **сразу с базой**, если тянет; иначе **закостылить** и дорисовать позже, сначала **добить базу** (**3–5 мин** не думает / код раздувается).

## Базовое решение (acceptance)

Без этого **не принять**:

1. Можно **положить** и **забрать** заказ.
2. Пользователю **уходит** сообщение с кодом (допускается первая версия **без** обработки ошибок API — см. грейд).
3. **Нельзя** положить **один заказ дважды**; **нельзя** дважды выдать по **одному коду**.
4. **Корректный стейт:** при **выдаче** ячейка **освобождается**.

Если сделана **только база** — выбор между **Hire II** и **Hire III** по таблице выше.

### Hire III по базе

- При ошибке **API** ячейка **не остаётся занятой** (откат брони).
- Если кандидат **не** предусмотрел — можно **подсказать**, что ошибки бывают; **штраф за подсказку не вешаем** (по методичке).
- Качество кода **не ниже 4**.

## Усложнение (acceptance)

- Шестизначный код, **без коллизий** в рамках постамата.
- **Криптографически разумный** рандом (**SecureRandom** и аналоги).
- **Не** счётчик/хеш заказа как «случайность».
- Ведущие нули **ок**.

## Пример базового решения (Kotlin)

Ниже — ориентир из методички. При использовании **проверьте**:

- при **`releaseCell`** нужно ли обнулять **`cellContainer[cellId]`**, чтобы ячейка реально стала **свободной** для следующего `putOrder`;
- **`getOrder`** по ТЗ должен ещё **вывести сообщение** пользователю (в примере часто возвращают только идентификатор ячейки).

```kotlin
class PostalBox(val api: Api, val cellsCount: Int) {
    private val cellContainer = Array<OrderData?>(cellsCount) { null }
    private val orderToCell = HashMap<String, Int>()
    private val codeToCell = HashMap<String, Int>()

    private val random = SecureRandom()

    fun putOrder(orderId: String) {
        val cellId = findFreeCell()

        if (orderToCell.containsKey(orderId)) {
            throw PostBoxException("Order already placed")
        }

        val code = generateCode()
        orderToCell[orderId] = cellId
        codeToCell[code] = cellId
        cellContainer[cellId] = OrderData(orderId, code)

        val apiResult = api.sendCode(orderId, code)
        if (!apiResult.ok) {
            releaseCell(cellId)
            throw PostBoxException("Failed to call api: ${apiResult.error}")
        }
    }

    fun getOrder(code: String): Int {
        val cellId = codeToCell[code]
            ?: throw PostBoxException("Invalid code passed")

        releaseCell(cellId)
        return cellId
    }

    private fun generateCode(): String {
        while (true) {
            val code = random.nextInt(100000, 999999).toString()
            if (!codeToCell.containsKey(code)) {
                return code
            }
        }
    }

    private fun findFreeCell(): Int {
        if (orderToCell.size >= cellsCount) {
            throw PostBoxException("Post box is full")
        }

        for (i in cellContainer.indices) {
            if (cellContainer[i] == null) {
                return i
            }
        }

        throw PostBoxException("Post box can't be placed: no free cell")
    }

    private fun releaseCell(cellId: Int) {
        val data = cellContainer[cellId] ?: return
        orderToCell.remove(data.orderId)
        codeToCell.remove(data.code)
    }
}

interface Api {
    fun sendCode(orderId: String, code: String): SendCodeResult
}

class OrderData(val orderId: String, val code: String)
class SendCodeResult(val ok: Boolean = true, val error: String? = null)

class PostBoxException(message: String) : RuntimeException(message)
```

## Tear-off: качество решения

### Придумывание решения — 0 / 0.5 / 1

Самостоятельность, API, расширяемость, тестируемость, структуры данных, асимптотика, corner cases.

- **0:** никакой самостоятельности; заметная помощь; неподходящие типы и не понимает после наводок.
- **0.5:** мелкие недочёты; рабочее, но неоптимально.
- **1:** без критичных проблем; оптимально или осознанно обосновано.

### Реализация — 0 / 1 / 2

Нейминг, читаемость, язык, оптимальность, валидация, ошибки API, **null**.

### Ошибки и их поиск — NH / 0 / 1 / 2

Как в общей методичке.

### Тестирование — 0 / 0.5 / 1

С **assert**; `main` + assert ок; принты без проверок — не засчитывать.

### Реализация усложнения

**1** балл при корректности без оставшихся багов.

### Не блокеры для hire, но давят на качество

- копипаста / громоздкое освобождение ячейки;
- слабая уникальность кодов;
- нет валидации, **NPE** на неизвестный код и т.п.;
- кандидат **не понимает** требования без помощи с придумыванием решения.
