# Оптимизация маршрута 2

## Условие

Дан **непустой** список направлений. Каждое направление — один символ:

| Символ | Направление |
|--------|-------------|
| `L` | Left |
| `R` | Right |
| `U` | Up |
| `D` | Down |

Каждый шаг сдвигает позицию на **1** в соответствующем направлении (на плоскости, старт из `(0, 0)` или другой фиксированной точки — главное быть последовательным).

**Петля** — ситуация, когда после выполнения части маршрута мы **снова оказываемся в уже посещённой** клетке. Такой фрагмент даёт **нулевой чистый эффект** относительно позиции «до захода в петлю» и считается **пустой тратой**; его нужно **убрать** из списка шагов.

**Важно:** цель **не** «добраться до конечной точки кратчайшим геометрическим путём». Нельзя **срезать** произвольной прямой там, где исходный маршрут так не ходил. Разрешена только **отмена завершённых петель** (возврат в ранее посещённую клетку).

Пример сохранения «формы» без петли:

```text
[D, R, U] -> [D, R, U]
```

### Примеры с удалением петель

```text
[R, D, L, U, R]     -> [R]
[R, D, L, R, U, U, R] -> [R, U, R]
```

**Ответ** — снова **массив направлений**, а не массив посещённых точек.

## Ограничения

- **Время:** `O(N)` по числу шагов `N`.
- **Память:** `O(N)` (дополнительно к входу).

## Что часто придумывают кандидаты (и почему это не идеально)

Типичная идея: класть в **hash map** текущие координаты → **индекс** в списке; при повторном заходе в клетку **вырезать** кусок от сохранённого индекса до текущего.

**Минусы:** легко ошибиться при удалении (случайно получить новую петлю, вырезать не то), забыть **почистить** map при вырезании, логика получается тяжёлой.

## Решение получше: visited + список координат

Держим **множество посещённых** координат и **список/стек** координат в порядке обхода. Как только текущая клетка **уже в set**, **откатываемся** по списку координат назад, **пока не выйдем** из «конфликтной» точки, и **по пути удаляем** координаты из set (и синхронно укорачиваем список шагов). Проще и быстрее по мысли, но **редко** придумывают без подсказки.

## Ещё вариант: map «из точки — куда идти дальше»

**Hash map:** координата → **следующее направление** (или следующий шаг) при «текущей» симуляции. Если снова попали в ту же клетку — **перезаписываем** исходящее направление новым.

**Тонкость (намекнуть, не бить):** если **финальная** точка всего маршрута лежит **на петле**, в map в этой клетке может остаться «куда идти дальше», хотя **дальше уже не нужно**. Сильный плюс, если кандидат **сам** это замечает и дорабатывает (например, отдельно хранить «терминальность» или финальный проход).

## Третий вариант: map + вектор посещённых точек

В **map** хранить **последний индекс** данной координаты в **векторе** посещённых точек (или в текущем построении маршрута); при обнаружении повтора — откат/усечение по индексу.

## О неоднозначности ответа

**Корректных** сокращённых маршрутов может быть **несколько** (например, при обработке **с конца** списка получится другой допустимый результат). **Не требовать** глобально **кратчайшую** длину среди всех допустимых — достаточно **любого** результата, удовлетворяющего правилам удаления петель.

## Ход интервью

- В начале полезно попросить **нарисовать** путь на листе для произвольной последовательности.
- Если всё прошло хорошо и есть время — обсудить **поиск оптимального** маршрута в смысле **shortest path** (уже другая, более тяжёлая постановка).
