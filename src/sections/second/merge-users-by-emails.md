# Слияние пользователей по списку email-ов

## Условие

Есть **n** пользователей; у каждого — **список email-адресов** (всего по всем пользователям **m** писем, одно и то же письмо может встречаться у нескольких пользователей).

**Правило:** если у двух пользователей есть **хотя бы один общий** email, это **один и тот же** пользователь (транзитивно: если A связан с B, а B с C, то A, B и C сливаются).

**Задача:** выполнить **слияние**. На выходе — список «склеенных» пользователей в том же формате (имя + список email-ов). **Имя** объединённого пользователя можно взять **любое** из исходных имён компоненты. В списке email-ов — только **уникальные** адреса.

**Параметры** `n` и `m` произвольны; длина списка email-ов у одного пользователя **не ограничена**.

### Пример

**Вход:**

```text
user1 -> xxx@ya.ru, foo@gmail.com, lol@mail.ru
user2 -> foo@gmail.com, ups@pisem.net
user3 -> xyz@pisem.net, vasya@pupkin.com
user4 -> ups@pisem.net, aaa@bbb.ru
user5 -> xyz@pisem.net
```

Связи: `user1`–`user2` (`foo@gmail.com`), `user2`–`user4` (`ups@pisem.net`), `user3`–`user5` (`xyz@pisem.net`).

**Выход** (один из допустимых вариантов имён):

```text
user1 -> xxx@ya.ru, foo@gmail.com, lol@mail.ru, ups@pisem.net, aaa@bbb.ru
user3 -> xyz@pisem.net, vasya@pupkin.com
```

## Источник

Близкая постановка на LeetCode: [Accounts Merge](https://leetcode.com/problems/accounts-merge/) (там аккаунты и имена; логика «общий email ⇒ один человек» та же).

## Желаемая сложность

**Линейная или близкая к линейной** по `n + m` (с учётом хеширования строк). Удобный инструмент — **DSU (Union–Find)** по пользователям: для каждого email объединить всех пользователей, у которых он встречается.

## Возможное решение (граф + компоненты)

1. Построить **отображение** `email → список индексов пользователей`, где встречается этот email (**hash map**), за **O(m)** с учётом вставок.
2. Для каждого email **соединить** пользователей из списка в одну компоненту: например, первого с вторым, второго с третьим, … (или через **union** в DSU) — **O(длина списка)** на email, суммарно **O(m)**.
3. Размер структуры порядка **O(n + m)**.
4. Найти **компоненты связности** (DFS/BFS по построенному графу или **find** в DSU для каждого пользователя).
5. Для каждой компоненты собрать **множество** всех email-ов её пользователей, выбрать **представителя** (любое имя), отсортировать email-ы при необходимости (в постановке LeetCode часто просят лексикографический порядок).

**Альтернатива:** не строить явно все рёбра, а только **DSU**: для каждого email выполнить `union(user_i, user_first)` для всех `i` в списке — асимптотика та же с path compression и union by rank.

## Что обсудить на интервью

- Транзитивность («цепочка» общих писем).
- Уникальность email-ов в ответе (**set** / **distinct**).
- Сложность при **длинных** списках у одного пользователя.
